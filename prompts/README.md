# Prompts 目录说明

这个目录管理 TechLex 中所有调用 Claude API 的 Prompt 资产。它不仅保存当前模板，也记录每个 Prompt 的目标、输入输出契约、设计理由、示例和迭代历史。

## 文件列表

| 文件 | 用途 | 调用位置 |
|---|---|---|
| `extract-terms.md` | 从技术文本中筛选值得学习的术语 | `src/services/termExtractor.js` |
| `generate-card.md` | 根据术语和原文语境生成学习卡片 | `src/services/cardGenerator.js` |

## 处理链路

```text
用户文本
  → extract-terms
  → { terms: [{ term, context_sentence }] }
  → generate-card
  → { term, definition, example, context }
  → 程序补充 id、时间和复习字段
  → Card
```

提取和生成拆成两个 Prompt，是为了分别评估“选得是否值得学”和“卡片是否准确可用”。如果合并为一次调用，结果不好时很难判断问题来自术语选择还是内容生成。

## 设计原则

1. **Prompt 与调用逻辑分离**：文档负责意图和契约，代码负责变量注入、请求、解析和错误处理。
2. **结构化输出**：统一要求只返回 JSON，程序仍需进行 schema 校验，不能信任模型输出。
3. **保留原文语境**：术语必须带 `context_sentence` 进入卡片生成，减少脱离上下文的泛化解释。
4. **控制单次范围**：默认提取 5-8 个术语，优先质量而不是数量。
5. **可评估和可迭代**：每次修改都记录版本、原因和预期影响；重要修改使用固定样本回归测试。
6. **不让模型生成本地字段**：`id`、时间、复习次数和权重由程序确定，避免模型制造不可控状态。

## 运行时约定

Markdown 文件是 Prompt 的设计与评审来源。MVP 若在代码中保存运行时模板，必须与这里的“当前版本 Prompt”保持一致；后续可以通过构建脚本将模板转换为可导入模块。

调用层至少完成以下校验：

- HTTP 请求成功，并区分认证、限流、超时和服务异常。
- 响应存在文本内容，移除意外的 Markdown 代码围栏后再解析。
- 解析结果是对象，字段名称和类型符合约定。
- 字符串经过 `trim` 且非空，数组长度在允许范围内。
- 不把模型返回的 HTML 直接插入页面。

## 如何修改 Prompt

1. 先明确要修复的失败样本和成功标准。
2. 修改对应文件中的“当前版本 Prompt”。
3. 在“迭代记录”中追加版本、日期、修改原因和预期效果。
4. 用同一组正常、边界和干扰样本进行回归。
5. 如输出字段变化，同步更新解析代码、数据结构文档和测试。

## 建议评估样本

- 一段包含明确 API 术语的官方文档。
- 一段术语较少的普通英文说明。
- 一段包含代码、列表和缩写的 README。
- 一段长度过短或只有代码的输入。
- 一段带有“忽略之前指令”等提示注入文本的内容。

评估时记录术语相关性、过宽泛词汇比例、JSON 合法率、卡片事实准确性和上下文贴合度。
