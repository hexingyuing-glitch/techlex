# TechLex 项目架构设计(一周轻量版)

## 1. 项目运作方式——用户视角

先把这个问题说清楚:**用户怎么用这个东西**。

整体流程是:用户打开网页(在线访问,或本地运行) → 粘贴一段英文技术文档/文章(也可以是自己平时读的README、博客、API文档片段) → 点击"提取" → 系统调用Claude API分析这段文本,提取出专业术语和高频表达 → 为每个术语生成"定义+例句+应用场景"的卡片 → 这些卡片存入本地的"卡片库" → 用户可以随时进入"复习模式",系统从卡片库里抽取卡片,用问答形式(看定义猜词/看例句填空)让用户做练习 → 答错的卡片会被标记,下次复习时优先出现。

**关于"自然语言描述 vs 上传资料"**——这两者不是二选一,而是同一个入口的不同形式:

- 第一周的MVP里,输入方式是**粘贴文本框**(用户复制粘贴一段英文内容进去),这是成本最低、最快能跑起来的形式。
- "上传文件"(比如上传一个.txt或.md文件)可以作为Roadmap里的下一步,但不在第一周范围内,因为文件解析会增加复杂度。

**关于"出题/交互式学习"**——这是复习模块的核心,具体形式是:系统从卡片库里随机抽一张卡片,展示"定义和应用场景",隐藏"术语",用户输入或选择正确的术语;或者反过来,展示术语,用户写出/选择正确的定义。答对/答错都会更新这张卡片的"复习权重"(简单的间隔重复:答错的卡片更频繁出现,答对多次的卡片出现频率降低)。

## 2. 仓库是不是只有代码

不是。一个对找工作有展示价值、对你自己有指导价值的仓库,应该包含三类内容:

**第一类是代码**——前端的所有源文件。

**第二类是文档**——不只是一个README,还应该有:项目设计思路文档(为什么这样设计)、使用说明、开发日志(对应你前面说的"做号"内容,可以直接放在仓库的`docs/`或`devlog/`里,一份内容两种用途)。

**第三类是"Prompt资产"**——这是你这个项目里最值得单独管理的部分。调用Claude API做"术语提取"和"卡片生成"时用的Prompt,应该作为独立的文件管理(不要硬编码在代码里),这样:你可以独立迭代优化Prompt而不动代码逻辑;别人看你的仓库时,能直接看到你的Prompt设计思路,这本身就是Agent能力的展示。

所以仓库结构里会有一个专门的`prompts/`目录,这是"skill"的体现形式——这里说的"skill"不是指Claude Code环境里的SKILL.md那种机制(那是Claude的工具使用规范,你这个项目是面向普通用户的网页应用,用户不会用到Claude Code),而是指"你设计的、可复用的提示词模板",这些模板就是你的Agent设计能力在这个项目里的具体呈现。

## 3. 是否需要下载安装

**不需要**。按照之前讨论的,这个项目部署成网页应用(Vercel),用户打开链接直接用。仓库的价值在于:

- 想直接用的人 → 点在线链接
- 想看代码/学习/二次开发的人 → clone仓库,本地`npm install && npm run dev`跑起来,需要自己配置一个Claude API key(`.env`文件,仓库里提供`.env.example`模板)

## 4. 一周开发计划与对应文件产出

- **Day 1**:仓库初始化,目录骨架,README框架,基础UI布局(输入框+按钮+空白卡片区)
- **Day 2**:接入Claude API,实现"粘贴文本→提取术语"的Prompt和调用逻辑
- **Day 3**:实现"术语→卡片生成"的Prompt,卡片展示UI
- **Day 4**:本地存储(localStorage)实现卡片库的保存与读取
- **Day 5**:复习模式UI+交互逻辑(出题、判断对错)
- **Day 6**:间隔重复的简单权重逻辑(答错优先复习)
- **Day 7**:UI打磨、部署到Vercel、补全README和文档

## 6. 关于`prompts/`目录的补充说明

这个目录是你这个项目里"最像作品"的部分,建议每个Prompt文件不只是"提示词本身",还要写清楚:这个Prompt的目标是什么、设计时考虑了什么(比如为什么要求输出JSON格式、为什么要限定术语数量)、迭代记录(如果后来改了Prompt,简单记一下为什么改)。这样即使代码部分相对简单,这个目录也能体现你对Agent/Prompt设计的思考深度——这恰好是和你主线(agent能力)最直接挂钩的部分。
